# 批判性数据分析 (Critical Data Analysis)

## 任务目标

对复杂数据进行系统化的分析处理。**核心特色**是通过**多Agent批判性交叉验证机制**（"幻觉防火墙"），在每一个关键步骤对分析过程和结果进行独立验证，彻底杜绝数据分析中最常见的AI幻觉问题：

| ❌ 常见幻觉 | ✅ 防火墙机制 |
|-----------|-------------|
| 捏造不存在的数据点和统计量 | momus + oracle 双重数值验证 |
| 编造/美化不存在的趋势模式 | deep 独立方法复现 |
| 统计方法的误用和误读 | prometheus 方法规划 + momus 审查 |
| 报告中的过度自信和幻觉性表述 | oracle 报告逐点审查 |
| 图表的误导性呈现 | momus 可视化审查 |
| 不存在的文献引用或数据来源 | 知识边界约束 + oracle 核查 |

## 核心架构：幻觉防火墙 (Hallucination Firewall)

```
数据输入
  │
  ▼
┌──────────────────────────────────────────────────────────────┐
│ Phase 1: 计划阶段 — 方法合理性防线                            │
│  prometheus(制定计划) → momus(批判性审查) → 通过则继续        │
│  ⚠️ 未通过 → 返回修订，直到 momus 认可方法设计                │
└──────────────────────────────────────────────────────────────┘
  │
  ▼
┌──────────────────────────────────────────────────────────────┐
│ Phase 2: 预处理阶段 — 数据质量防线                            │
│  hephaestus(执行预处理) → momus(审查数据真实性) → 通过则继续  │
│  ⚠️ 发现数据编造痕迹 → 标记并修正                            │
└──────────────────────────────────────────────────────────────┘
  │
  ▼
┌──────────────────────────────────────────────────────────────┐
│ Phase 3: 分析阶段 — 数值正确性防线                            │
│  hephaestus(执行分析) → momus(审查结果合理性) → 通过则继续    │
│  ⚠️ 数值异常/过度完美 → 返回修正                              │
└──────────────────────────────────────────────────────────────┘
  │
  ▼
┌──────────────────────────────────────────────────────────────┐
│ Phase 4: 交叉验证 — 核心幻觉防火墙（并行）                     │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                   │
│  │ oracle   │  │ momus    │  │ deep     │                   │
│  │ 统计验证  │  │ 深度审查  │  │ 独立复现  │                  │
│  └──────────┘  └──────────┘  └──────────┘                   │
│  → 三路结果汇总 → 一致通过/存在差异/验证失败                  │
└──────────────────────────────────────────────────────────────┘
  │
  ▼
┌──────────────────────────────────────────────────────────────┐
│ Phase 5: 报告阶段 — 表述准确性防线                            │
│  writing(撰写报告) → oracle(逐点审查) → 修正 → 最终输出       │
│  ⚠️ 每个数据点必须有真实来源，不确定性必须如实传达              │
└──────────────────────────────────────────────────────────────┘
  │
  ▼
  最终分析报告（含完整验证记录）
```

## 执行流程

### Phase 1: 数据理解与计划

#### 1.1 初始数据探查

```bash
# 预览数据结构和前 N 行
head -n 20 {data_path}
# 或使用 Python 快速探查
python3 -c "
import pandas as pd
df = pd.read_csv('{data_path}')
print('Shape:', df.shape)
print('Columns:', df.columns.tolist())
print('Dtypes:')
print(df.dtypes)
print('\\nMissing values:')
print(df.isnull().sum())
print('\\nBasic stats:')
print(df.describe())
"
```

理解：变量类型、缺失比例、分布特征、潜在的数据质量问题。

#### 1.2 制定分析计划

委托 `prometheus` 制定详细分析计划，任务描述包含：

```
数据路径: {data_path}
分析目标: {analysis_goal}
数据摘要: [上一步的输出]

请制定分析计划，包含:
1. 数据质量检查方案（缺失值、异常值、一致性）
2. 核心分析方法与统计模型（含方法选择的理由）
3. 可视化方案
4. 量化评估指标
5. 潜在的混淆变量和局限性
6. 验证策略（如何确保结论稳健）

计划必须保守诚实，避免过度承诺。
```

#### 1.3 计划审查（防火墙 checkpoint #1）

委托 `momus` 严格审查分析计划：

```
审查以下分析计划，重点关注:
1. 方法选择是否合理？有无统计误用风险？
2. 是否存在潜在的幻觉点（可能编造的趋势、过度解释的风险）？
3. 计划是否可执行且结论范围恰当？
4. 局限性是否被充分承认？
5. 有无遗漏重要的验证步骤？

【分析计划】
{上一步的输出}

请给出: 通过/需修改/拒绝 的判断及具体理由。
```

**通过条件**：momus 判定"通过"或仅需轻微修改。否则返回 1.2 修订计划。

---

### Phase 2: 数据预处理与探索性分析

#### 2.1 执行预处理与 EDA

委托 `hephaestus` 按照审核通过的计划编写并运行代码：

```
请为以下分析目标编写完整的数据预处理和探索性分析代码:

数据路径: {data_path}
分析目标: {analysis_goal}
审核通过的计划: {Phase 1 的输出}

要求:
1. 代码必须设置随机种子 (seed=42)
2. 每一步打印中间结果（缺失值处理、异常值处理等）
3. 生成基础统计摘要
4. 生成探索性可视化（保存到 {output_dir}/figures/）
5. 包含数据质量报告
6. 代码保存到 {output_dir}/code/eda.py
```

#### 2.2 预处理审查（防火墙 checkpoint #2）

委托 `momus` 审查预处理过程和结果：

```
审查以下数据预处理结果:

1. 缺失值处理方法是否引入偏倚？
2. 异常值剔除是否有统计依据？
3. 数据转换是否合理（正态性检验等）？
4. 有没有"过于完美"的数据特征（幻觉迹象）？
5. 数据质量报告中是否有警示信号？

【预处理代码】: {output_dir}/code/eda.py
【预处理输出】: [上一步的 stdout/stderr]

请给出: 通过/需修改/拒绝 的判断及具体理由。
```

**通过条件**：momus 未发现数据编造迹象或不当处理方法。

---

### Phase 3: 核心分析执行

#### 3.1 执行核心分析

委托 `hephaestus` 按照审核通过的计划实施分析：

```
请根据审核通过的分析计划，编写核心分析代码:

数据路径: {data_path}
分析目标: {analysis_goal}
预处理结果目录: {output_dir}/
审核通过的计划: {Phase 1 的输出}

【核心分析要求】
1. 每一步打印中间结果（便于验证复现）
2. 随机种子固定 (seed=42)
3. 统计检验报告: 效应量 + 置信区间 + p值（不过度依赖p值）
4. 数值结果的合理性检查（数量级、边界检查 → 防幻觉）
5. 模型评估: 训练/测试分离 + 交叉验证
6. 代码保存到 {output_dir}/code/analysis.py
7. 结果保存到 {output_dir}/results.json
```

#### 3.2 分析审查（防火墙 checkpoint #3）

委托 `momus` 审查核心分析结果：

```
审查以下分析结果的合理性和正确性:

1. 数值量级检查 — 结果是否在合理范围内？（防数字幻觉）
2. 统计假设是否满足（正态性、方差齐性、独立性等）？
3. 是否存在 p-hacking、多重比较未校正等问题？
4. 效应量是否实际显著（区分统计显著与实际显著）？
5. 不确定性是否被如实报告？
6. 代码逻辑是否有明显错误？

【分析代码】: {output_dir}/code/analysis.py
【分析结果】: {output_dir}/results.json
【控制台输出】: [上一步的 stdout/stderr]

请给出: 通过/需修改/拒绝 的判断及具体理由。
```

**通过条件**：momus 判定通过或仅需轻微修正。

---

### Phase 4: 多角度交叉验证（核心幻觉防火墙）

**此阶段并行运行三个独立验证**，从不同角度交叉检验分析结论。

#### 4.1 Oracle 独立统计验证

委托 `oracle` 进行独立的统计验证：

```
请独立验证以下分析的正确性（不使用已有的分析代码）:

数据路径: {data_path}
分析目标: {analysis_goal}
分析结果: {output_dir}/results.json

请以统计专家的角度:
1. 关键统计量是否合理？（独立心算/估算验证数量级）
2. 方法选择是否最优？有无更好的替代方法？
3. 结果解释是否忠于数据？有无过度解读？
4. 如果发现可疑之处，明确标记

输出到: {output_dir}/validation/oracle_verification.md
```

#### 4.2 Momus 深度幻觉检测

委托 `momus` 进行专项幻觉检测：

```
请进行深度幻觉检测，重点关注:

1. 是否存在编造的数据点或统计量？
2. 结果是否"过于完美"（如 R²=0.99 在真实数据中极罕见）？
3. 不确定性是否被如实报告（置信区间是否合理宽度）？
4. 图表是否存在误导性（坐标轴截断、选择性的数据展示等）？
5. 结论是否超出了数据所能支持的范围？

【分析结果】: {output_dir}/results.json
【图表目录】: {output_dir}/figures/

输出到: {output_dir}/validation/momus_review.md
```

#### 4.3 Deep 独立方法复现

委托 `deep` 使用不同的方法独立复现关键结论：

```
请用【不同的分析方法】独立复现核心结论，验证结果稳健性:

数据路径: {data_path}
分析目标: {analysis_goal}
原分析方法: [Phase 1 计划中的方法]

要求:
- 如果原分析用了参数检验（t检验/ANOVA），请用非参数方法（Mann-Whitney/Kruskal-Wallis）
- 如果原分析用了线性模型，请用树模型/非参数方法
- 如果原分析用了频率统计，请用贝叶斯方法
- 对比两种方法的结果差异，评估结论的一致性

输出结果对比到: {output_dir}/validation/deep_replication.md
```

#### 4.4 验证结果汇总

汇总三个验证结果：

```markdown
## 幻觉防火墙验证报告

### 各维度状态

| 验证维度 | 验证者 | 状态 | 关键发现 |
|---------|--------|:----:|---------|
| 统计验证 | oracle | ⬜ | 待汇总 |
| 深度审查 | momus | ⬜ | 待汇总 |
| 独立复现 | deep | ⬜ | 待汇总 |

### 总体判定
- ✅ **一致通过** — 结论可靠（所有验证均通过）
- ⚠️ **存在差异** — 以下方面需要进一步讨论: [具体条目]
- ❌ **验证失败** — 存在潜在幻觉/错误，返回 Phase 3 修正
```

**判定规则**：
- 三个验证全部通过 → ✅ 继续 Phase 5
- 一个验证有轻微异议 → ⚠️ 记录争议点，继续 Phase 5 但在报告中标注
- 一个验证有严重问题或多个验证有异议 → ❌ 返回 Phase 3 修正

委托 `writing` 撰写综合数据分析报告：

请撰写综合数据分析报告，基于以下所有材料:

分析目标: {analysis_goal}

## 注意事项

### 1. 幻觉防火墙优先级
验证结果的**优先级高于分析结果**。如果任何验证点发现问题：

```
发现幻觉/错误
  → 立即中止当前 Phase
  → 标记具体问题（哪个文件、哪行、什么数值）
  → 返回对应 Phase 修正
  → 修正后重新运行所有下游验证
```

不可在已知存在幻觉的情况下继续推进。

### 2. Agent 职责速查

| Agent | 角色 | 在本 Skill 中的职责 |
|-------|------|-------------------|
| **prometheus** | 战略规划 | 制定数据驱动的分析计划，确保方法合理性 |
| **hephaestus** | 稳健实现 | 编写可验证、可复现的分析代码 |
| **momus** | 批判性审查 | 在每个 checkpoint 识别幻觉、逻辑谬误、统计误用 |
| **oracle** | 独立验证 | 从零开始验证统计正确性，审查报告准确性 |
| **deep** | 替代复现 | 用不同方法验证结论稳健性 |
| **writing** | 专业撰写 | 生成结构清晰、表述准确的报告 |

### 3. 可复现性规范
- 所有分析代码设置 `random.seed(42)` / `np.random.seed(42)`
- 记录 Python/packages 版本（生成 requirements.txt）
- 每一步保存中间结果（拒绝仅保存最终结果）
- 图表的坐标轴、标题、图例完整标注

### 4. 知识边界约束
本 skill 严格遵守 [知识边界规范](../knowledge_boundary.md)。在数据分析中特别关注：

- **不编造统计量**：不得补充分析未计算出的统计值
- **不确定的因果推断**：必须明确标注为"相关性"而非"因果"
- **未能验证的数据**：标注"未验证"并说明原因
- **不美化效应量**：如实报告效应大小，不人为放大
- **诚实面对负面结果**：不显著的结果也是有效结果，不得掩盖

### 5. 数据隐私保护
- 包含敏感信息的数据，脱敏后再交由 agent 处理
- 不在 agent 调用中传递原始敏感字段
- 聚合统计（均值、中位数等）通常安全，个体级数据谨慎处理

### 6. 失败处理
如果以下情况发生，如实报告并终止：

- 数据无法读取或损坏
- 核心假设无法满足（无有效方法可用）
- 多个验证者持续发现不可修复的问题
- 数据量过小无法支持可靠结论
